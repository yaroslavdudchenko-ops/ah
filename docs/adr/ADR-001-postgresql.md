# ADR-001: PostgreSQL 16 как база данных

**Status:** Accepted | **Date:** 2026-04-23 | **Version:** 1.0.0 | **Author:** System Architect

---

## Контекст

Сервис хранит протоколы клинических исследований, каждый из которых состоит из набора текстовых секций с произвольной структурой. Ключевое требование — версионирование на уровне секций и diff между версиями. Данные нужны надёжные и персистентные между деплоями.

## Рассмотренные варианты

| Вариант | За | Против |
|---|---|---|
| **PostgreSQL 16** | JSONB для гибкой структуры секций; транзакции; надёжность; named volumes для Dokploy | Отдельный контейнер в compose |
| **SQLite** | Нет отдельного сервиса; простота | Нет JSONB; слабый concurrent write; file-based — риск потери при деплое без явного volume |
| **MongoDB** | BSON нативен для документов | Избыточен для MVP; нет транзакций в простых конфигурациях; нет в стандартных образах |

## Решение

Выбран **PostgreSQL 16-alpine**.

## Обоснование (Rationale)

Контент секций протокола (`introduction`, `objectives`, `design`, ...) хранится в поле `content JSONB` таблицы `protocol_versions`. JSONB позволяет:
- добавлять новые секции без миграции схемы
- делать diff между версиями на уровне Python (`dictdiffer` / `difflib`)
- использовать транзакции при сохранении версии + open issues

Доступ через SQLAlchemy 2 (async) + asyncpg. Миграции — Alembic.

## Последствия

- В `docker-compose.yml` добавлен сервис `db` с named volume `db-data`
- `DATABASE_URL` — обязательная переменная окружения
- При деплое на Dokploy `db-data` volume сохраняется между деплоями
