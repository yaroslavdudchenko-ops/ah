## 1. Контекст

### 1.1. Домен

Фармацевтическая компания полного цикла. Клинические исследования — ключевой процесс разработки лекарственных препаратов, регулируемый международными стандартами GCP (Good Clinical Practice) и ICH (International Council for Harmonisation).

### 1.2. Проблема

Протокол клинического исследования — документ объёмом 50–200 страниц, критичный по научной корректности и юридической точности. Текущий процесс его разработки:

| Проблема | Следствие |
|---------|-----------|
| Команды копируют старые протоколы вручную | Переносятся «хвосты» от старых дизайнов |
| Нет автоматической проверки терминологии | Разночтения препарат/МНН в разных разделах |
| Нет контроля структуры | Пропускаются обязательные разделы ICH E6 |
| Нет автоматической проверки логики | Противоречия между конечной точкой и планом анализа |

### 1.3. Стейкхолдеры

| Роль | Интерес |
|------|---------|
| Научный сотрудник (Clinical Researcher) | Быстро получить черновик протокола по параметрам |
| Медицинский ревьюер | Проверить GCP-соответствие, получить список вопросов |
| Регуляторный специалист | Уверенность в структурной полноте документа |
| CTO / ML Lead | Демонстрация AI-возможностей компании |

## 3. Бизнес-требования

### BR-01 — Библиотека шаблонов протокола `P1`

**Обоснование:** Различные фазы КИ имеют принципиально разные дизайны и структурные требования.

| ID | Требование |
|----|-----------|
| BR-01.1 | Система предоставляет готовые шаблоны для Фаз I, II и III |
| BR-01.2 | Поддерживаются 4 типа дизайна: рандомизированный, открытый (open-label), плацебо-контролируемый, двойной слепой |
| BR-01.3 | Шаблон определяет промпты для всех 9 разделов протокола |
| BR-01.4 | Исследователь выбирает шаблон при создании нового протокола |

---

### BR-02 — Ввод ключевых параметров `P0`

**Обоснование:** Качество AI-генерации прямо зависит от полноты и структурированности входных данных.

| ID | Требование |
|----|-----------|
| BR-02.1 | Система принимает: название препарата, МНН, терапевтическую область |
| BR-02.2 | Система принимает описание целевой популяции пациентов |
| BR-02.3 | Система принимает первичную и вторичные конечные точки |
| BR-02.4 | Система принимает длительность исследования и схему дозирования |
| BR-02.5 | Система принимает структурированные критерии включения и исключения |
| BR-02.6 | Все обязательные поля валидируются до запуска генерации |

---

### BR-03 — AI-генерация разделов протокола `P0`

**Обоснование:** Основная ценность сервиса — автоматическое создание структурированного научного текста.

| ID | Требование |
|----|-----------|
| BR-03.1 | Система генерирует 9 разделов в соответствии со структурой ICH E6 R2 |
| BR-03.2 | Генерируемый текст учитывает специфику терапевтической области и фазы КИ |
| BR-03.3 | Каждый раздел имеет структурированные заголовки, не является сплошным текстом |
| BR-03.4 | Исследователь может перегенерировать отдельный раздел без потери остальных |
| BR-03.5 | Система информирует исследователя о прогрессе генерации в реальном времени |

---

### BR-04 — Контроль консистентности `P1`

**Обоснование:** Главная причина ошибок — несоответствие терминологии и логические противоречия между разделами.

| ID | Требование |
|----|-----------|
| BR-04.1 | Система выявляет разночтения в написании одного термина по всему документу (препарат/МНН) |
| BR-04.2 | Система обнаруживает логические противоречия между разделами: конечная точка ↔ план анализа, популяция ↔ критерии исключения, длительность ↔ график визитов |
| BR-04.3 | Каждое несоответствие классифицируется по типу и критичности (высокая / средняя / низкая) |
| BR-04.4 | Система предоставляет рекомендации по устранению противоречий со ссылкой на ICH E6 |
| BR-04.5 | Результат проверки включает числовой GCP-скор соответствия (0–100) |
| BR-04.6 | Проверка не подменяет юридическую или медицинскую экспертизу — об этом явно предупреждает UI |

---

### BR-05 — Связанные артефакты `P2`

**Обоснование:** Полный пакет КИ включает не только протокол, но и сопровождающие документы.

| ID | Требование |
|----|-----------|
| BR-05.1 | По запросу исследователя система генерирует заготовку SAP (Statistical Analysis Plan) |
| BR-05.2 | По запросу исследователя система генерирует заготовку ICF (Informed Consent Form) |
| BR-05.3 | Артефакты экспортируются в форматах Markdown и DOCX |

---

### BR-06 — Версионирование `P1`

**Обоснование:** Протокол проходит множество итераций ревью — необходима история изменений.

| ID | Требование |
|----|-----------|
| BR-06.1 | Каждое сохранение автоматически создаёт пронумерованную версию (v0.1, v0.2, ..., v1.0) |
| BR-06.2 | Исследователь может добавить текстовый комментарий к версии |
| BR-06.3 | История версий доступна в UI с датой и комментарием |
| BR-06.4 | Сравнение двух версий отображает изменения на уровне разделов `P2` |
| BR-06.5 | Любую предыдущую версию можно открыть в режиме просмотра |

---

### BR-07 — Экспорт `P0/P2`

**Обоснование:** Итоговый документ должен интегрироваться в существующие регуляторные процессы.

| ID | Требование | Приоритет |
|----|-----------|-----------|
| BR-07.1 | Экспорт в Markdown с сохранением иерархии заголовков | P0 |
| BR-07.2 | Экспорт в HTML с навигацией по якорям (#section-name) | P0 |
| BR-07.3 | Экспорт в DOCX со стандартными стилями заголовков | P2 |
| BR-07.4 | «Список открытых вопросов» для медицинского ревьюера | P2 |
| BR-07.5 | Экспортируемый файл содержит метаданные: версия, дата, фаза, препарат | P0 |
| BR-07.6 | Все экспорты синтетических данных содержат водяной знак | P0 |
| BR-07.7 | AI-контент помечен: `"AI-Assisted. Requires qualified person review."` | P0 |

## 4. Ограничения и допущения

### 4.1. Ограничения

| # | Допущение |

| A-03 | Сеть имеет доступ к внутреннему AI Gateway (корпоративная инфраструктура) |
