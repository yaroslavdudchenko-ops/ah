## FR-02 Ввод ключевых параметров

*Источник: corecase.md §2 | Приоритет: P0*

| ID | Требование | Приоритет | Источник |
|---|---|---|---|
| FR-02.1 | Форма ввода принимает: индикацию (терапевтическую область), название препарата, МНН | P0 | corecase §2 |
| FR-02.2 | Форма принимает описание популяции пациентов (свободный текст, max 500 символов) | P0 | corecase §2 |
| FR-02.3 | Форма принимает первичную конечную точку (primary endpoint) и вторичные конечные точки (secondary endpoints, список строк) | P0 | corecase §2 + расширение |
| FR-02.4 | Форма принимает длительность исследования **в неделях** (диапазон 1–520; пример: 96 недель ≈ 22 месяца) и схему дозирования | P0 | corecase §2 |
| FR-02.5 | Критерии исключения вводятся структурированно (список строк, добавление/удаление элементов) | P0 | corecase §2 |
| FR-02.6 | Все поля валидируются перед отправкой: `phase IN ('I','II','III')`, `duration_weeks > 0`, `primary_endpoint` непустой (min 3 символа), `title` непустой (min 5 символов) | P0 | corecase §2 |
| FR-02.7 | Форма принимает критерии включения (Inclusion Criteria) как список строк (GCP ICH E6 R2 §4.3) | P0 | ICH E6 §6.1 |

## FR-03 AI-генерация разделов

*Источник: corecase.md §3, GCP ICH E6 структура | Приоритет: P0*

| ID | Требование | Приоритет | Источник |
|---|---|---|---|
| FR-03.1 | Система генерирует 12 разделов (GCP ICH E6 / GCP ЕАЭС): (1) Title Page, (2) Synopsis, (3) Introduction & Background, (4) Study Objectives, (5) Study Design, (6) Study Population, (7) Study Treatment, (8) Efficacy Assessments, (9) Safety Assessments, (10) Statistical Analysis, (11) Ethics, (12) References. MVP минимум — 7 разделов (3–9) | P0 | corecase §3 + ICH E6 + rf-protocol-sections-reference |
| FR-03.2 | Каждый раздел генерируется отдельным вызовом к AI Gateway (InHouse/Qwen3.5-122B) с передачей входных параметров протокола в качестве контекста. Использование внешних LLM запрещено | P0 | corecase §3 + политика ИБ |
| FR-03.3 | Генерация адаптирует язык и акценты под терапевтическую область и фазу (`therapeutic_area_context`, `phase_context`) | P0 | corecase AI-требования |
| FR-03.4 | Текст структурирован с явными подзаголовками H2/H3, не представляет собой сплошной текст | P0 | corecase AI-требования |
| FR-03.5 | Пользователь может перегенерировать отдельный раздел без пересоздания всего протокола | P0 | corecase §3 |
| FR-03.6 | Время генерации полного протокола (9 секций параллельно) не превышает 120 секунд при стабильном соединении | P0 | corecase §3 |

## FR-04 Контроль консистентности

*Источник: corecase.md §4, AI-требования | Приоритет: P1*

| ID | Требование | Приоритет | Источник |
|---|---|---|---|
| FR-04.1 | Система обнаруживает ≥ 2 формы написания одного термина (препарат/МНН) методом нечёткого сравнения (fuzzy matching, порог схожести ≥ 85%) по всему документу | P1 | corecase §4 |
| FR-04.2 | Система выявляет логические противоречия между разделами: endpoint vs. план анализа, populace vs. критерии исключения, длительность vs. график визитов | P1 | corecase §4 |
| FR-04.3 | Результат проверки отображается как список несоответствий с полями: тип, severity (high/medium/low), затронутые разделы, описание, рекомендация | P1 | corecase §4 |
| FR-04.4 | Система предлагает GCP-compliance подсказки с указанием нормативной ссылки (ICH E6 R2 §X) | P1 | corecase AI-требования |
| FR-04.5 | Ответ содержит `compliance_score` (0–100) — числовая оценка GCP-соответствия версии | P1 | расширение |
| FR-04.6 | Проверка консистентности не подменяет юридическую или медицинскую экспертизу; UI отображает явный дисклеймер | P1 | безопасность |

## FR-05 Связанные артефакты

*Источник: corecase.md §5 | Приоритет: P2*

| ID | Требование | Приоритет | Источник |
|---|---|---|---|
| FR-05.1 | По запросу пользователя система генерирует заготовку SAP (Statistical Analysis Plan — план статистического анализа, ~20-40 стр.) | P2 | corecase §5 |
| FR-05.2 | По запросу пользователя система генерирует заготовку ICF (Informed Consent Form — форма информированного согласия пациента, требование GCP ЕАЭС §4.8 / 61-ФЗ) | P2 | corecase §5 |
| FR-05.3 | Артефакты SAP/ICF экспортируются отдельно в форматах Markdown и DOCX. Интеграция с внешними ERP/BI системами не предусмотрена | P2 | corecase §5 |
| FR-05.4 | Артефакты версионируются вместе с основным протоколом (хранятся в `protocol_versions.content` JSONB под ключами `sap` / `icf`) | P2 | расширение |
| FR-05.5 | SAP заготовка содержит минимальную структуру: гипотезы, популяции анализа (ITT/PP/Safety), первичная и вторичные конечные точки, методы работы с пропущенными данными | P2 | rf-protocol-guide §17 |
| FR-05.6 | ICF заготовка содержит минимальную структуру: цели, процедуры, риски/польза, добровольность, конфиденциальность, контакты | P2 | rf-protocol-guide §21 + Приказ №353н |

## FR-07 Экспорт

*Источник: corecase.md §7 | Приоритет: P0 (MD+HTML), P2 (DOCX)*

| ID | Требование | Приоритет | Источник |
|---|---|---|---|
| FR-07.1 | Экспорт в Markdown (.md) с сохранением иерархии заголовков H1/H2/H3 | P0 | corecase §7 |
| FR-07.2 | Экспорт в HTML с корректным рендерингом и навигацией по разделам (якоря #section-name) | P0 | corecase §7 |
| FR-07.3 | Экспорт в DOCX по стандарту OOXML (ISO/IEC 29500): H1→Heading1, H2→Heading2, H3→Heading3 | P2 | corecase §7 |
| FR-07.4 | Экспорт «Списка открытых вопросов» для медицинского ревьюера отдельным файлом | P2 | corecase §7 |
| FR-07.5 | Экспортируемый файл содержит метаданные: версия протокола, дата генерации, фаза, препарат | P0 | расширение |
| FR-07.6 | Все экспорты синтетических данных содержат водяной знак: `"FOR DEMONSTRATION PURPOSES ONLY — SYNTHETIC DATA"` | P0 | правило 01-security |
| FR-07.7 | AI-сгенерированный контент помечается меткой `"AI-Assisted. Requires qualified person review."` | P0 | GCP ICH E6 R2 |

## Нефункциональные требования

*Источник: corecase.md §Constraints | Приоритет: P0*

| ID | Требование | Приоритет | Источник |
|---|---|---|---|
| NFR-01 | Веб-интерфейс отображается корректно в Chrome, Firefox, Edge (последние 2 мажорные версии) | P0 | corecase constraints |
| NFR-02 | API отвечает на запросы (кроме AI-генерации) не дольше 2 секунд при стандартной нагрузке (1 пользователь) | P0 | corecase constraints |
| NFR-03 | Приложение запускается командой `docker compose up --build` без дополнительных зависимостей | P0 | corecase constraints |
| NFR-04 | Секреты (API ключи, пароли) не коммитятся в репозиторий; используется `.env` файл | P0 | правило 01-security |
| NFR-05 | Все AI-вызовы логируются в `audit_log` с полями: timestamp, model, duration_ms, section_type | P0 | правило 01-security |
| NFR-06 | Отклонение реализованного интерфейса и функциональности от данного документа не должно превышать **15%** для стандартных функций и **25%** для AI-генератора. Отклонение = (число нереализованных/сломанных FR-пунктов) / (общее число FR-пунктов в области) × 100%. Косметические UI-различия (цвет, отступы, иконки) в расчёт не входят. | P0 | правило 06-development-rules |
| NFR-07 | **Требования законодательства РФ:** (а) AI-генерируемые протоколы содержат русскоязычные разделы — обязательно; (б) GCP-валидатор проверяет соответствие: 61-ФЗ, Приказу №353н, GCP ЕАЭС (Решение ЕЭК №79), ГОСТ Р 52379-2005, ICH E6 R2; (в) Документация хранится ≥ 10 лет (требование Приказа №353н); (г) UI содержит ссылку на Федеральный реестр КИ Минздрава РФ для регистрации исследования | P1 | 61-ФЗ + Приказ №353н + rf-protocol-guide |
| NFR-08 | **Политика ИБ — только локальные модели:** данные протоколов КИ (содержащие потенциально конфиденциальную медицинскую и коммерческую информацию) передаются **исключительно** во внутренний AI Gateway. Любые вызовы к внешним LLM-сервисам (OpenRouter, OpenAI, Anthropic и др.) запрещены в production-коде | P0 | role-infosec + политика ИБ |
